# testgen-llm-loop

Meta TestGen-LLM 論文（"Automated Unit Test Improvement using Large Language Models at Meta", 2024）の知見を、Claude Code / Cursor などのエージェント環境で再現するための反復ループスキル。

「LLM にテストを書かせる」だけならエージェントは追加コンテキスト無しでも一定の出力を出すが、それでは TestGen-LLM 論文が指摘する 3 つの失敗パターン（ビルドが通らない、カバレッジを増やさない、同じ失敗テストを繰り返し提案する）に陥る。このスキルはそれを構造的に防ぐためにある。

## 基本原則

このスキルは以下の前提に立つ。逸脱しないこと。

1. **既存スイートの「拡張」のみを行う**。テストクラス全体の再構築・テストフレームワーク変更は本スキルの範囲外（論文と同じ制限）。
2. **採否はカバレッジで機械的に判定する**。「良さそうに見えるテスト」を主観で残さない。
3. **採用されなかったテストは次反復のプロンプトに戻す**。何が落ちたか、なぜ価値がなかったかを LLM が次に知れる状態にする。
4. **生成→検証は同一エージェントセッション内で完結させる**。検証を別パスに切ると、失敗テストを「あったこと」として LLM が忘れる。

## ループ全体像

```
[Phase 1] 対象選定        ── どのファイルにテストを足すかを絞る
   ↓
[Phase 2] コンテキスト収集 ── 既存テストのスタイル、カバレッジ穴、規約
   ↓
[Phase 3] ベースライン確立 ── 現状の test command / coverage を確実に動く状態にする
   ↓
[Phase 4] テスト生成       ── 既存スタイルに合わせて N 件提案
   ↓
[Phase 5] 4 基準フィルタ   ── compile / run / coverage / uniqueness で篩う
   ↓
[Phase 6] flakiness check ── 採用候補を複数回実行して安定性を確認
   ↓
[Phase 7] フィードバック   ── 不採用テストを次反復の負例として記録
   ↓
[Phase 8] PR 化            ── 採用テストだけを既存スイートに統合し PR を開く
```

カバレッジ目標到達（即停止）or 最大反復回数到達 or 収束 で停止。Phase 4 〜 Phase 7 はループ。

## Phase 1: 対象選定

「リポジトリ全体をやる」ことは提案しない。論文の運用でも対象は絞られている。

### 0. リポジトリ前提チェック（最優先）

対象選定の前に必ず実施する：

1. 対象リポジトリを最新化（`fetch --prune` 相当）する。
2. VCS CLI または VCS REST API で、本スキルが作成する想定のフィーチャブランチ・進行中 PR（同一目的の testgen 作業）が既に存在しないか確認する。
3. 既存ブランチ・PR が見つかった場合、Phase 8 のどのフローで進めるかを **この時点で** 確定する：
   - **新規 PR**: 別名ブランチを切る、または既存ブランチを整理してから新規作成
   - **既存 PR への追記**: 既存ブランチ上に採用テストのみ追加
   - **検証のみで終了**: 既存 PR の内容と今回の生成が同一なら変更を作らず、検証結果のみ報告

### 対象選定の優先順位

次の優先順位で対象を確定する：

1. ユーザーが具体ファイル/ディレクトリを指定 → それを使う
2. PR コンテキスト（変更ファイルがある）→ 変更ファイルのうちカバレッジが下がる/穴があるものを使う
3. 指定がない → カバレッジレポートを見て「カバレッジが閾値未満かつ最近変更頻度が高い」ファイルを 1〜3 件提案し、ユーザー確認を取る

対象が決まったら、対象ファイルとそれに対応する既存テストファイルのパス、リポジトリで使われているテストランナー、カバレッジレポートの出力形式をリポジトリから確認する（設定ファイル・既存 CI・ドキュメントを参照）。

確認が取れない場合は推測で進めない。ユーザーに聞く。

## Phase 2: コンテキスト収集

LLM 単発で精度を上げる最大のレバーは「既存テストのスタイルを真似させること」。次を必ず収集する。

- **対象ソースの中身**：関数シグネチャ、戻り値、依存（DI、外部呼び出し、I/O）
- **同じファイル / 隣接ファイルの既存テスト 3〜5 件**：命名規約、アサーション粒度、モック方針、フィクスチャ
- **テスト共通設定**：conftest.py, setup ファイル, テストヘルパー
- **カバレッジ穴**：行カバレッジ / 分岐カバレッジで現状ヒットしていない箇所を特定（このスキルは「未カバー部分を埋める」ことが目的なので、すでにカバーされている経路に対するテストは原則作らない）

これらを Phase 4 のプロンプトに「Reference style」「Uncovered branches」「Project conventions」として明示的に区分けして渡す。混ぜて渡すと LLM はスタイルを無視しがち。

## Phase 3: ベースライン確立

ここでサボると後段が全て無駄になる。

ベースライン環境は **実行場所（local / CI / リモート実行環境など）を問わない**。次の 2 条件を満たすこと：

1. 既存テストスイートがそのまま実行され、**グリーン**であること
2. カバレッジレポートが **1 コマンド（または固定した短い手順）で再現可能**に生成できること（Phase 5 で毎反復繰り返すため）

手順：

1. 既存テストスイートをそのまま実行 → グリーンであることを確認
2. カバレッジレポートを生成 → ファイル/行レベルで読める形にする
3. 上記を再現するコマンド列を固定し、以降の反復で同じ手順を使う

既存スイートがレッド or 不安定なら、生成テストの採否判定が不可能になる。その場合は先にユーザーに状況を報告し、生成フェーズに進まない。

## Phase 4: テスト生成

プロンプトに含めるべき要素は `references/prompt-patterns.md` を参照。

要点：

- 1 反復で複数件（3〜10 件）まとめて提案させる。1 件ずつ生成するとフィードバックのコストが嵩む。
- 「未カバー分岐 X を埋めること」を明示する。漠然と「テストを増やせ」と言わない。
- 既存のテストファイルに追記する形で出力させる。新規テストクラスを勝手に作らせない。
- 次反復以降は、過去の不採用テストとその不採用理由を「Failed test candidates (do not regenerate equivalent tests)」セクションとして渡す。

## Phase 5: 4 基準フィルタ

論文の核心。詳細は `references/filter-criteria.md`。各テストを次の順番で篩う。1 つでも落ちたら破棄。

1. **Build**: 当該ファイルだけでなく、追加後のテストファイル全体がコンパイル/import 解決できるか
2. **Pass**: 当該テスト単体が green になるか
3. **Coverage gain**: 追加によりカバレッジが**増加するか**（既存テストと同じ行/分岐しか踏まないなら破棄）
4. **Uniqueness**: 同一反復内の他の採用候補と実質的に重複していないか（assertion がほぼ同じ、入力値だけ違うが同じ分岐を踏む等）

破棄したテストは「不採用理由」とともに記録する。Phase 7 で使う。

## Phase 6: flakiness check

採用候補に残ったテストは、テストの性質に応じた回数だけ連続実行し、全てパスすることを確認する。1 回でも落ちたら flaky と判定し、その時点で破棄する。

論文では **5 回連続**が提案されている。これは安全側の**上限**とする。リスク評価が困難な場合はデフォルトで 5 回を維持する。

| リスク | 該当する性質（いずれか） | 連続実行回数 |
|--------|--------------------------|--------------|
| 低 | I/O / 並列 / 時間依存 / ネットワーク / 非決定的依存のいずれも持たない | 1 回 |
| 中 | 上記のうち 1 つを持つ | 3 回 |
| 高 | 上記のうち 2 つ以上を持つ、または並列実行で順序依存が出る | 5 回 |

並列実行で順序依存のフレークが出る場合があるため、高リスクと判定したテストでは可能ならテスト並列度を変えた実行も含める。

## Phase 7: フィードバック蓄積

このフェーズを省略すると論文の TestGen-LLM ではなく「ただの LLM ループ」になる。

破棄されたテストごとに、次の構造で記録する：

```
- test name: <生成されたテスト名>
- reason for rejection: build_error | test_failure | no_coverage_gain | duplicate | flaky
- artifact: <コンパイル/実行エラーメッセージ or カバレッジ判定の根拠 or 重複先テスト名>
```

次反復の Phase 4 に渡すプロンプトには、この一覧を「再生成しない不採用テスト」として含める。同じ理由で落ちるテストが繰り返し提案される現象は、論文・Qodo Cover 実装の両方が報告している既知の問題で、これを抑える唯一の確実な方法は履歴をプロンプトに戻すこと。

## Phase 8: PR 化

採用テストだけを既存テストファイルに統合し、VCS CLI または VCS REST API でプルリクエスト（マージリクエスト等、ホスティングの用語に合わせる）を開く。

### Phase 1 で既存ブランチ・PR があると判定された場合

Phase 1 で確定したフローに従う：

- **新規 PR**: 既存ブランチを別名にして再作成する、または既存ブランチを整理してから新規作成する
- **既存 PR への追記**: 既存ブランチを fetch し、rebase / merge のうちリポジトリ慣行に合う方で整合したうえ、採用テストのみを追加コミットして push する
- **検証のみで終了**: 既存 PR の内容が今回の生成と同一なら新規変更を作らず、四基準フィルタ・flakiness・カバレッジ推移の検証結果のみレポートに記載する

### 共通ルール

- 1 PR にまとめるテスト数の上限を設ける（推奨 5〜10 件、レビュー負荷の制御）。超える場合は分割
- PR の description には次を含める
  - 対象ファイルと変更前後のカバレッジ
  - 採用テスト一覧と各テストが埋めたカバレッジ穴の対応
  - 不採用テスト数と内訳（参考情報，人間レビュー時の判断材料）
  - 停止理由（カバレッジ目標到達 / 最大反復 / 収束 / ベースライン破損）

直接 main（デフォルトブランチ）へコミットするのは原則しない。理由は「追従しているだけで実装の意図を検証していないテスト」が混入したときに人間レビューで止めるため。

---

## 停止条件

次の優先順位で判定し、**最初に該当した時点で停止**する：

1. **カバレッジ目標到達 → 即停止**（その反復で採用テストが 1 件以上あっても、目標到達なら次反復に進まない）
2. 最大反復回数（推奨 3〜5）に到達
3. 反復しても新規採用テストが 0 件の状態が 2 反復連続（収束）
4. ベースラインが壊れた（実行環境の問題）

停止時、現在の採用テスト一覧と未到達のカバレッジ穴を報告する。

## 範囲外（明示的にやらないこと）

- 統合テスト、E2E テスト、性能テストの生成（論文のスコープ外）
- 既存テストの修正・リファクタリング（拡張のみ）
- カバレッジ穴の検出に基づかない「念のためのテスト」追加
- ユーザーが指定していないファイルへの自発的な拡張
- mutation testing による品質判定（カバレッジは「行を踏んだ」を意味するが「assertion が意味を持つ」は保証しない。この限界は把握しておくこと）
- カバレッジ目標到達後に assertion の意味を検証したい場合は、対象言語で確立している mutation testing ツール（各言語エコシステムに OSS が存在する）へ切り替え、**別ループ**として実施する。本スキルの範囲外とする

## 出力するもの

- 採用テスト（既存テストファイルへの追記 or 新規テストファイル）
- PR（VCS REST API 経由、または PR description のドラフト）
- 反復ごとのレポート（採用/不採用件数、カバレッジ推移、不採用理由内訳）
